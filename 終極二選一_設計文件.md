> 版本：v2.0 | 日期：2026-04-20 | 更新：Supabase Realtime → Socket.io 架構遷移

## 一、專案概覽

| 項目 | 說明 |
|------|------|
| 遊戲名稱 | 懂我再說 |
| 玩法 | 每回合一位「被猜者」選主題 → 系統出題（A/B）→ 被猜者私下選答 → 其他人猜測 → 全員提交後揭曉 |
| 即時同步 | ~~Supabase Realtime~~ → **Socket.io（Node.js 後端）** |
| 前端部署 | GitHub Pages |
| 後端部署 | Railway / Render（Node.js + Socket.io） |
| 身份識別 | localStorage + UUID（匿名，刷新後身份保留） |

## 二、技術選型

| 層級 | 技術 | 原因 |
|------|------|------|
| 前端 | 純 HTML / CSS / Vanilla JS | 無需建置工具，GitHub Pages 直接部署 |
| 樣式 | Tailwind CSS (CDN) | 快速排版，無需打包 |
| 即時同步 | **Socket.io（取代 Supabase Realtime）** | 保證事件送達，無掉訊問題，支援 Server→Client push |
| 後端 | **Node.js + Express + Socket.io** | 管理 in-memory 房間狀態、廣播遊戲事件 |
| 資料庫 | Supabase PostgreSQL | 持久化房間/玩家/回合資料；後端以 `service_role` key 存取 |
| 身份識別 | localStorage + `crypto.randomUUID()` | 無需登入，刷新後可重連 |
| 房間 ID | nanoid 6 碼英數 (大寫) | 短、可讀、易分享 |

### 2.1 為何從 Supabase Realtime 遷移到 Socket.io

**原始問題**：Supabase Realtime 基於 PostgreSQL Logical Replication，事件**不保證送達**。  
遊戲中最常見的卡住情境：

| 情境 | 問題 |
|------|------|
| 房主選完題目後其他玩家卡住 | `rounds` 的 UPDATE 事件被 Realtime 丟棄，其他玩家的訂閱沒收到通知 |
| 玩家重新整理後無法重新加入 | Realtime 訂閱在重連後有競爭條件，事件到達順序不穩定 |
| 大廳有 polling 補救，遊戲中沒有 | 只有大廳有 15 秒 polling fallback；選題/選答/猜測階段一旦掉訊就無法繼續 |

**Socket.io 方案（Option B）優點**：
- Server 主動 push，不依賴 DB 訂閱
- `io.to(roomId).emit()` 廣播給同一房間所有人，保證送達
- Server 端持有 `roomCache`（in-memory），無需每次查 DB
- 斷線重連時 Server 重送完整 `room_state`，前端自動恢復

## 三、頁面架構

```
index.html          ← 首頁：建立房間 / 輸入房間碼加入
room.html?id=XXXX   ← 房間大廳 + 遊戲主畫面（SPA-like 狀態切換）
```

### 3.1 index.html

- **建立房間區塊**
  - 輸入暱稱
  - 輸入房間人數上限（2～12）
  - 按下「建立房間」→ 產生房間 → 跳轉 `room.html?id=XXXX`

- **加入房間區塊**
  - 輸入房間碼
  - 按下「加入」→ 跳轉 `room.html?id=XXXX`

### 3.2 room.html（多狀態切換）

依 `room.status` 與 `round.status` 切換以下畫面區塊（用 JS show/hide）：

| 狀態 | 顯示畫面 |
|------|----------|
| `waiting` | 大廳畫面 |
| `playing` + `selecting_topic` | 選主題畫面 |
| `playing` + `selecting_answer` | 被猜者選答畫面 |
| `playing` + `guessing` | 其他玩家猜測畫面 |
| `playing` + `revealing` | 揭曉結果畫面 |
| `finished` | 遊戲結束畫面 |

#### 大廳畫面元件
- 房間碼 + 一鍵複製連結按鈕
- 玩家列表（暱稱、是否房主、連線狀態）
- 準備按鈕（非房主）
- 開始遊戲按鈕（房主，需全員準備）
- 踢人按鈕（房主）

#### 選主題畫面
- 提示「你是本回合被猜者」（僅被猜者看到操作）
- 主題卡片選擇（事業 / 愛情 / 獵奇 / …）
- 非被猜者看到「等待 XXX 選主題…」

#### 被猜者選答畫面
- 顯示本回合題目：A 選項 vs B 選項
- 被猜者點選 A 或 B（私下提交，其他人只看到「等待 XXX 作答…」）

#### 猜測畫面
- 顯示題目（A / B）
- 其他玩家點選 A 或 B
- 顯示「已提交 X / N 人」進度條
- 被猜者看到「其他人猜測中…」

#### 揭曉畫面
- 大字顯示正確答案（A 或 B）
- 每位玩家猜測結果（✓ / ✗）
- 猜對人數統計
- 「下一回合」按鈕（房主）

#### 遊戲結束畫面
- 所有玩家回合統計
- 「再玩一次」按鈕（房主重置房間）

## 四、Supabase 資料庫 Schema

### 4.1 topics（主題）

```sql
CREATE TABLE topics (
  id   UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name TEXT NOT NULL UNIQUE  -- '事業', '愛情', '獵奇', ...
);
```

### 4.2 questions（題目）

```sql
CREATE TABLE questions (
  id       UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  topic_id UUID REFERENCES topics(id) ON DELETE CASCADE,
  option_a TEXT NOT NULL,
  option_b TEXT NOT NULL
);
```

### 4.3 rooms（房間）

```sql
CREATE TABLE rooms (
  id                TEXT PRIMARY KEY,           -- 'ABC123' 6碼
  host_player_id    TEXT NOT NULL,              -- localStorage uuid
  max_players       INT  NOT NULL DEFAULT 8,
  status            TEXT NOT NULL DEFAULT 'waiting',
  -- status: waiting | playing | finished
  current_round_id  UUID,                       -- 外鍵指向 rounds
  created_at        TIMESTAMPTZ DEFAULT NOW(),
  updated_at        TIMESTAMPTZ DEFAULT NOW()
);
```

### 4.4 players（玩家）

```sql
CREATE TABLE players (
  id          TEXT PRIMARY KEY,           -- localStorage uuid
  room_id     TEXT REFERENCES rooms(id) ON DELETE CASCADE,
  nickname    TEXT NOT NULL,
  is_ready    BOOLEAN NOT NULL DEFAULT FALSE,
  is_online   BOOLEAN NOT NULL DEFAULT TRUE,
  join_order  INT NOT NULL,               -- 決定輪流順序
  score       INT NOT NULL DEFAULT 0,
  created_at  TIMESTAMPTZ DEFAULT NOW(),
  updated_at  TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(room_id, nickname)               -- 同房間不可重複暱稱
);
```

### 4.5 rounds（回合）

```sql
CREATE TABLE rounds (
  id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  room_id           TEXT REFERENCES rooms(id) ON DELETE CASCADE,
  round_number      INT NOT NULL,
  subject_player_id TEXT REFERENCES players(id),  -- 被猜者
  topic_id          UUID REFERENCES topics(id),
  question_id       UUID REFERENCES questions(id),
  subject_answer    TEXT,   -- 'A' | 'B'，選完前為 NULL
  status            TEXT NOT NULL DEFAULT 'selecting_topic',
  -- status: selecting_topic | selecting_answer | guessing | revealing | finished
  created_at        TIMESTAMPTZ DEFAULT NOW(),
  updated_at        TIMESTAMPTZ DEFAULT NOW()
);
```

### 4.6 guesses（猜測）

```sql
CREATE TABLE guesses (
  id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  round_id   UUID REFERENCES rounds(id) ON DELETE CASCADE,
  player_id  TEXT REFERENCES players(id),
  guess      TEXT NOT NULL,    -- 'A' | 'B'
  is_correct BOOLEAN,          -- 揭曉後填入
  created_at TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(round_id, player_id)  -- 每人每回合只能猜一次
);
```

### 4.7 used_questions（已使用題目，避免重複）

```sql
CREATE TABLE used_questions (
  room_id     TEXT REFERENCES rooms(id) ON DELETE CASCADE,
  question_id UUID REFERENCES questions(id),
  PRIMARY KEY (room_id, question_id)
);
```

## 五、Socket.io 即時事件設計（取代 Supabase Realtime）

### 5.1 架構總覽

```
前端 (GitHub Pages)                   後端 (Railway/Render)
index.html                             Node.js + Express
  └─ 建立/加入房間（Supabase 直接寫入）         │
                                              │
room.html ←──── Socket.io ────────────────→ server.js
  socket-client.js                         roomCache (Map)
  room-page.js                           + Supabase (service_role)
```

### 5.2 前端 → 後端 事件（Client Emit）

| 事件 | Payload | 說明 |
|------|---------|------|
| `join_room` | `{ roomId, playerId, nickname }` | 進入房間，取得完整狀態 |
| `toggle_ready` | `{ roomId, playerId, ready }` | 切換準備狀態 |
| `start_game` | `{ roomId, playerId }` | 房主開始遊戲 |
| `select_topic` | `{ roomId, playerId, topicId }` | 被猜者選主題 |
| `submit_answer` | `{ roomId, playerId, answer }` | 被猜者私下提交答案 |
| `submit_guess` | `{ roomId, playerId, guess }` | 其他玩家提交猜測 |
| `reveal_round` | `{ roomId, playerId }` | 房主手動揭曉（全猜完會自動揭曉） |
| `next_round` | `{ roomId, playerId }` | 房主進入下一回合 |
| `end_game` | `{ roomId, playerId }` | 房主提前結束遊戲 |
| `restart_game` | `{ roomId, playerId }` | 房主重新開始 |
| `kick_player` | `{ roomId, hostId, targetId }` | 房主踢人 |
| `set_online` | `{ isOnline }` | 頁面可見性變更時通知 |

### 5.3 後端 → 前端 事件（Server Emit）

| 事件 | Payload | 說明 |
|------|---------|------|
| `room_state` | `{ room, players, topics, currentRound, currentQuestion, guesses }` | 加入成功後的完整快照 |
| `join_error` | `{ message }` | 加入失敗（房間不存在、已滿等） |
| `players_updated` | `{ players }` | 玩家列表異動 |
| `game_started` | `{ room, currentRound }` | 遊戲開始，進入第一回合 |
| `round_updated` | `{ currentRound, currentQuestion }` | 回合狀態流轉（選題→選答→猜測） |
| `guess_progress` | `{ submitted, total }` | 即時更新猜測進度條 |
| `round_revealed` | `{ currentRound, guesses, players }` | 揭曉結果（含完整答案與猜測） |
| `game_finished` | `{ room, players }` | 遊戲結束 |
| `game_restarted` | `{ room, players }` | 重新開始 |
| `answer_accepted` | `{ answer }` | 僅通知被猜者答案已收到 |
| `you_were_kicked` | — | 通知被踢出的玩家 |
| `error` | `{ message }` | 一般錯誤訊息 |

### 5.4 Server 端 in-memory 快取（roomCache）

```js
// roomCache: Map<roomId, CacheEntry>
{
  room:            {},   // rooms 資料列
  players:         [],   // players 陣列
  topics:          [],   // 可選主題
  currentRound:    {},   // 當前 rounds 資料列
  currentQuestion: {},   // 當前題目（選答/猜測階段）
  guesses:         [],   // 本回合猜測（揭曉前 answer 遮蔽）
  _revealing:      false // 防止 auto-reveal 重複觸發鎖
}
```

> **subject_answer 保護**：Server 端 `buildRoundPayload(round, forReveal)` 在 `forReveal=false` 時遮蔽 `subject_answer`，前端猜測階段永遠看不到正確答案。

### 5.5 自動揭曉機制

所有「有資格猜測的玩家」（排除被猜者、離線玩家）全部提交猜測後，Server 自動觸發 `revealRound()`，無需房主手動操作。  
使用 `cache._revealing` 布林鎖防止同時兩個玩家提交觸發兩次揭曉。

## 六、遊戲狀態機（同 v1.0）

```
[房間狀態]
  waiting
    ↓ 房主按「開始遊戲」（全員已準備）
  playing
    ↓ 所有玩家輪完
  finished

[回合狀態]（每回合一個 rounds 資料列）
  selecting_topic
    ↓ 被猜者選好主題 → 系統抽題
  selecting_answer
    ↓ 被猜者私下選 A/B
  guessing
    ↓ 其他玩家全員提交猜測
  revealing
    ↓ 房主按「下一回合」 or 自動計時
  finished
    ↓ 若還有玩家未當被猜者 → 建立新 round
    ↓ 若全員輪完 → room.status = 'finished'
```

## 七、前端狀態管理（localStorage + 記憶體）

```js
// 本地持久化
localStorage.setItem('player_id', uuid);     // 玩家唯一 ID
localStorage.setItem('nickname', '...');      // 上次使用暱稱

// 記憶體狀態（頁面載入後從 Supabase fetch）
state = {
  room: { id, status, host_player_id, current_round_id, max_players },
  players: [],               // 依 join_order 排序
  currentRound: {},          // rounds 資料
  guesses: [],               // 本回合所有 guesses
  myPlayerId: '',            // 來自 localStorage
  _renderTimer: null,        // render 防抖計時器（50ms，合併短時間多次呼叫）
  _playersDebounceTimer: null, // players 變更防抖計時器（250ms）
}

// render 防抖機制：
// 多個 Realtime handler 在 50ms 內連續觸發 render() 時，只執行最後一次 _doRender()
// 避免同一批事件（onRoundChange + onRoomUpdate）造成畫面重複繪製與閃爍
```

## 八、防呆機制

| 情境 | 處理方式 |
|------|---------|
| 暱稱重複 | DB UNIQUE 約束 → 前端顯示錯誤提示 |
| 斷線重連 | 進入房間時用 player_id 查詢 players，若存在則 UPDATE is_online=true |
| 房間不存在 | 查詢 rooms 失敗 → 導回 index.html 並提示 |
| 房間已滿 | 查詢 players count >= max_players → 阻擋加入 |
| 房主離線 | is_online=false 但不轉移房主；房主重連後恢復控制 |
| 房間無人 | 定時/trigger 偵測 players 全數 is_online=false → 刪除/封存房間 |
| 題庫用盡 | 抽題時若同主題已無未使用題目 → 提示換主題或重置已用題目 |
| **畫面閃爍** | `showSection()` 記錄當前顯示的 section id，相同 section 直接跳過，不重跑切換動畫 |
| **Realtime 多事件重繪** | `render()` 加入 50ms 防抖，短時間內多次呼叫只執行最後一次，消除連續事件造成的畫面跳動 |
| **題目不同步（INSERT 亂序）** | `onRoundChange` 收到 INSERT 事件時，若 round id 與現有相同（UPDATE 已先設好題目），保留 `currentQuestion` 不清空 |
| **題目不同步（stale 讀）** | `onRoomUpdate` 執行 await 後二次確認 `currentRound.id`，若 `onRoundChange` 已先完成更新則跳過，避免 stale 資料覆蓋正確題目 |

## 十、可擴充預留點

| 功能 | 預留方式 |
|------|---------|
| 計分制 | `players.score` 欄位已建立，揭曉時批次 UPDATE |
| 更多主題 | `topics` 表可隨時新增 |
| 限時猜測 | `rounds` 可加 `guess_deadline TIMESTAMPTZ` 欄位 |
| 房主踢人 | `players` DELETE by id（已有 cascade） |
| 跳過回合 | `rounds.status` 直接設 `finished`，建立下一 round |
| 觀眾模式 | `players` 加 `role TEXT DEFAULT 'player'`（player/spectator） |

```env
SUPABASE_URL=https://xxxx.supabase.co
SUPABASE_SERVICE_KEY=eyJ...   # ⚠️ 必須用 service_role key，非 anon key
CORS_ORIGIN=http://localhost:5500,http://127.0.0.1:5500
```

> **⚠️ 重要**：`SUPABASE_SERVICE_KEY` 必須填入 **service_role key**（非 anon key）。  
> 取得位置：Supabase 後台 → Project Settings → API → **service_role** secret。

### 11.2 本機開發啟動

```bash
# 後端（Terminal 1）
cd server
node server.js          # 啟動於 http://localhost:3000

# 前端（Terminal 2）
npx serve . -p 5500     # 啟動於 http://127.0.0.1:5500
```

> **`serve.json` 說明**：專案根目錄有 `serve.json` 設定 `"cleanUrls": false`。  
> 新版 `serve`（v14+）預設會將 `/room.html?id=XXX` 重導向到 `/room`（移除 `.html` 並**丟棄 query string**），導致 room-page.js 讀不到 `roomId` 而跳回首頁。  
> `cleanUrls: false` 停用此行為，確保 query string 正確傳遞。

## 十二、Supabase RLS 建議

```sql
-- 所有人可讀房間、玩家、回合（遊戲需要）
ALTER TABLE rooms   ENABLE ROW LEVEL SECURITY;
ALTER TABLE players ENABLE ROW LEVEL SECURITY;
ALTER TABLE rounds  ENABLE ROW LEVEL SECURITY;
ALTER TABLE guesses ENABLE ROW LEVEL SECURITY;

-- 簡易匿名策略（搭配 anon key）
CREATE POLICY "public read" ON rooms   FOR SELECT USING (true);
CREATE POLICY "public read" ON players FOR SELECT USING (true);
CREATE POLICY "public read" ON rounds  FOR SELECT USING (true);
CREATE POLICY "public read" ON guesses FOR SELECT USING (true);

-- 寫入策略（依業務規則收緊，初期可全開）
CREATE POLICY "public write" ON rooms   FOR ALL USING (true);
CREATE POLICY "public write" ON players FOR ALL USING (true);
CREATE POLICY "public write" ON rounds  FOR ALL USING (true);
CREATE POLICY "public write" ON guesses FOR ALL USING (true);
```

> ⚠️ 初期開發用全開策略，上線前應根據 `player_id` 收緊寫入權限。

## 十三、已知問題與解決記錄

| 問題 | 根本原因 | 解決方式 |
|------|----------|---------|
| 房主選完主題後其他人卡住 | Supabase Realtime 事件不保證送達，遊戲階段無 polling 補救 | 遷移至 Socket.io（v2.0） |
| 點「建立房間」後頁面跳回首頁（無任何 log） | `npx serve` v14+ 預設啟用 Clean URLs，將 `/room.html?id=XXX` 301 重導向至 `/room`（query string 遺失） | 新增 `serve.json` 設定 `"cleanUrls": false` |
| DevTools Console Preserve log 沒有留住 log | 由於上述 301 redirect 在 JS 執行前就發生（純 HTTP 層），JS 完全沒機會寫 log | 同上，修 serve 設定即可 |
