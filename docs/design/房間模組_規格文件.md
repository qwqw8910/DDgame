## 1. Objective

### 背景

目前平台有兩個多人遊戲：

| 遊戲 | 房間方式 | 命名空間 | 主要問題 |
|------|---------|---------|---------|
| 懂我再說 | Socket.io 主 namespace | `io` (default) | 斷線/重連邏輯碎片化；踢人設計有但程式碼未實作 |
| 默契傳聲筒 | Socket.io `/character-storm` | `io.of('/character-storm')` | 無踢人；server 重啟後 `role`/`quota` 消失；一律禁止中途加入 |

兩套遊戲的「建立房間、加入房間、踢人、斷線重連」邏輯各自實作、無法共用。

### 目標

**抽出獨立的 `server/room/` 模組**，提供：

1. 統一生命週期管理
2. 斷線後遊戲繼續、重連後補發狀態
3. 觀戰模式（無法中途加入時的 UX 替代方案）
4. 房主踢人（含最低人數保護）
5. 房主自動轉移
6. 各遊戲保留自己的邏輯（房間模組只管「誰在房間」）

### 使用者故事

- 身為玩家，我斷線後重新整理，還能回到原本的遊戲
- 身為觀戰者，我掃 QR Code 進房間時遊戲已開始，我可以先看，下一局自動成為玩家
- 身為房主，我可以踢出不活躍的玩家，遊戲繼續進行
- 身為房主，我關掉頁面後，房主自動轉移，其他人不受影響
- 身為開發者，新增遊戲只需掛載房間模組並實作 hooks，不重寫房間邏輯

## 5. 現況分析與問題

### 問題一：踢人功能缺失

| 遊戲 | 狀態 |
|------|------|
| 懂我再說 | 設計文件有「踢人按鈕」，`server.js` **未實作** `kick_player` handler |
| 默契傳聲筒 | 完全無踢人設計 |

### 問題二：server 重啟後重連失敗（默契傳聲筒）

- `role`、`quota` 只存在 in-memory cache
- server 重啟後 cache 消失，重連玩家拿不回角色
- **修復**：遊戲開始時將角色分配持久化至 `character_storm_sessions`

### 問題三：兩套 join 邏輯無法共用

- 懂我再說：`join_room` event
- 默契傳聲筒：`cs:join` event
- **修復**：統一改為 `room:join`，不做向下相容（前後端同步切換）

## 6. 中途加入 + 觀戰模式設計

### 6.1 各遊戲中途加入規則

#### 懂我再說

| 房間狀態 | 能否加入 | 說明 |
|---------|---------|------|
| `waiting` 大廳 | ✅ 正常加入 | — |
| `playing` 任意階段 | ✅ 加入為玩家 | 若在 `guessing` 階段，本輪跳過猜測 |
| `settling` 結算 | ✅ 正常加入 | 下局直接參與 |

新玩家的 `join_order` 排在現有所有玩家之後；「被猜順序」也依此排在最後。

#### 默契傳聲筒

| 房間狀態 | 能否加入 | 說明 |
|---------|---------|------|
| `waiting` 大廳 | ✅ 正常加入 | — |
| `playing` 單輪進行中（round1/round2 輸入 or 猜題） | ❌ | 改為**觀戰** |
| `playing` revealing 空檔（等房主按下一位） | ✅ 加入為玩家 | 下一輪才分配角色 |
| `settling` 結算 | ✅ 正常加入 | 下局直接參與 |

### 6.2 觀戰模式（Spectator）

當玩家無法中途加入，系統改為讓其「觀戰」，體驗如下：

```
使用者掃 QR / 貼連結進房間
  │
  ├─ 後端判斷：目前允許加入為正式玩家？
  │
  ├─ YES → 正式玩家（room:join-ack { type: 'player' }）
  │
  └─ NO → 觀戰者（room:join-ack { type: 'spectator' }）
               │
               ├─ 收到 room:join-ack（含目前遊戲公開快照）
               ├─ 畫面顯示：「遊戲進行中，你正在觀戰 👀 下一輪將自動加入」
               ├─ 持續收到公開廣播：倒數/提示進度/猜題結果
               │
               └─ hooks.spectatorCanUpgrade() 回傳 true 時
                    └─ 廣播 room:spectator-can-join → 顯示「加入下一輪」按鈕
                         └─ 玩家點擊 → emit room:spectator-join
                              └─ 轉換為正式玩家，排在玩家列表尾端
```

**觀戰者特性**：

| 特性 | 說明 |
|------|------|
| 不在 `players` 陣列 | 有獨立 `spectators` 列表 |
| 不計入人數上限 | `max_players` 只限正式玩家 |
| 只收公開資訊 | 不收被猜者私密答案等 |
| 無需重新掃碼/輸入資訊 | 直接升級為玩家 |

### 6.3 加入流程統一圖

```
emit room:join { roomId, nickname, playerId }
  │
  ├─ playerId 已存在 players 列表 → 重連（type: 'reconnect'）
  ├─ 新使用者 + 允許加入 → 正式玩家（type: 'player'）
  └─ 新使用者 + 不允許加入 → 觀戰者（type: 'spectator'）
       └─ hooks.spectatorCanUpgrade() = true 時
            └─ emit room:spectator-join → 升為正式玩家
```

## 7. 房間模組功能規格

### 7.1 房間生命週期

```
created
  │
  ▼
waiting（大廳）← 可加入為玩家
  │
  ▼ start_game
playing（遊戲中）← 由 hooks 決定能否加入為玩家；否則觀戰
  │
  ▼ all_rounds_done
settling（結算）← 觀戰者自動升為玩家
  │
  ▼ 房主解散 or 空房間 30 分鐘無人重連
archived（封存）
```

### 7.2 玩家狀態機

```
(未連線)
  │
  ▼ room:join（允許加入，或重連）
online ──斷線──→ offline ──重連──→ online
  │
  ├──room:leave──→ left（移除）
  └──room:kick───→ kicked（移除）

(未連線)
  │
  ▼ room:join（不允許加入）
spectating ──room:spectator-join──→ online（正式玩家）
```

**各狀態行為**：

| 狀態 | 在玩家列表 | 收到事件 | 可操作遊戲 |
|------|-----------|---------|-----------|
| online | ✅ | 全部（含私密） | ✅ |
| offline | ✅（標記離線） | 重連後補發 | ✗（待重連） |
| spectating | ❌（在觀戰列表） | 公開事件 | ✗ |
| left / kicked | ❌ | 無 | ✗ |

### 7.3 Socket 事件規格

#### Client → Server

| 事件 | 參數 | 說明 |
|------|------|------|
| `room:create` | `{ roomId, nickname, playerId, maxPlayers, appId, options? }` | 建立房間 |
| `room:join` | `{ roomId, nickname, playerId }` | 加入（或重連，或觀戰） |
| `room:leave` | — | 主動離開 |
| `room:kick` | `{ targetPlayerId }` | 房主踢人 |
| `room:transfer-host` | `{ targetPlayerId }` | 房主主動轉移 |
| `room:ready` | `{ ready: boolean }` | 切換準備（大廳） |
| `room:spectator-join` | — | 觀戰者升級為玩家 |

#### Server → Client

| 事件 | 接收方 | 參數 | 說明 |
|------|--------|------|------|
| `room:join-ack` | 本人 | `{ type, room, players, spectators, myPlayerId, gameState? }` | 加入確認＋完整快照 |
| `room:player-joined` | 房間+觀戰 | `{ player }` | 新玩家加入 |
| `room:player-reconnected` | 房間+觀戰 | `{ playerId }` | 玩家重連 |
| `room:player-offline` | 房間+觀戰 | `{ playerId }` | 玩家斷線 |
| `room:player-left` | 房間+觀戰 | `{ playerId, reason }` | 玩家離開 |
| `room:players-updated` | 房間+觀戰 | `{ players }` | 玩家列表更新 |
| `room:spectator-joined` | 房間+觀戰 | `{ spectator }` | 觀戰者加入 |
| `room:spectator-can-join` | 觀戰者 | — | 通知現在可升為玩家 |
| `room:spectators-updated` | 房間+觀戰 | `{ spectators }` | 觀戰列表更新 |
| `room:host-changed` | 房間+觀戰 | `{ newHostId }` | 房主轉移 |
| `room:kicked` | 被踢者 | `{ reason? }` | 你被踢了 |
| `room:error` | 請求者 | `{ code, message }` | 錯誤 |
| `room:closed` | 房間+觀戰 | — | 房間關閉 |

### 7.4 房主轉移規則

| 觸發條件 | 行為 |
|---------|------|
| 房主主動 `room:transfer-host` | 立即轉移給指定玩家 |
| 房主主動 `room:leave` | 轉移給 join_order 最小的在線玩家 |
| 房主斷線超過 **30 秒** | 轉移給 join_order 最小的在線玩家 |
| 房間只剩 1 位正式玩家 | 該玩家自動成為房主 |
| 所有玩家都離線 | `suspended` 狀態；任意玩家重連時自動轉移 |

### 7.5 踢人規則

```
房主 emit room:kick { targetPlayerId }
  │
  ├─ 驗證：請求者 = hostId？ → 否：room:error FORBIDDEN
  ├─ 驗證：不可踢自己 → 否：room:error CANNOT_KICK_SELF
  ├─ 驗證：踢出後玩家數 >= hooks.minPlayers？
  │    → 否：room:error KICK_BELOW_MIN（含說明：「至少需 N 人，踢出後剩 M 人」）
  │
  ├─ 傳送 room:kicked → 被踢者
  ├─ 斷開被踢者 socket（socket.disconnect()）
  ├─ 移除玩家列表
  ├─ 廣播 room:player-left { reason: 'kick' }
  ├─ 廣播 room:players-updated
  └─ 呼叫 hooks.onPlayerLeft(roomId, playerId, 'kick')
```

**minPlayers 由各遊戲定義**：

| 遊戲 | minPlayers |
|------|-----------|
| 懂我再說 | 2 |
| 默契傳聲筒 | 4 |

### 7.6 斷線不中斷遊戲

| 離開方式 | 保留玩家 | 角色保留 | 遊戲繼續 |
|---------|---------|---------|---------|
| offline（斷線） | ✅ 標記離線 | ✅ | ✅ 倒數到期後系統跳過 |
| left（主動離開） | ❌ 移除 | ✗ | ✅ hooks 跳過角色 |
| kicked（被踢） | ❌ 移除 | ✗ | ✅ hooks 跳過角色 |

### 7.7 重連補發（State Replay）

重連時後端依序補充：

1. `room:join-ack { type: 'reconnect' }` — 完整房間快照
2. `hooks.onPlayerJoined(socket, roomId, playerId, isReconnect=true)` — 遊戲自身補發
   - 默契傳聲筒：補角色、提示進度、剩餘計時
   - 懂我再說：補當前回合狀態、已提交猜測

### 7.8 Hooks API

```javascript
/**
 * @typedef {Object} RoomHooks
 *
 * @property {number} [minPlayers=2]
 *   最低玩家數（低於此數拒絕踢人）
 *   懂我再說=2，默契傳聲筒=4
 *
 * @property {(roomId: string) => boolean} [canJoinAsPlayer]
 *   新使用者是否能加入為正式玩家
 *   false → 改為觀戰者
 *
 * @property {(roomId: string) => boolean} [spectatorCanUpgrade]
 *   觀戰者是否可升為玩家
 *   true → 後端廣播 room:spectator-can-join
 *
 * @property {(roomId: string) => object|null} [buildGameState]
 *   填入 room:join-ack 的 gameState 欄位
 *
 * @property {(socket, roomId, playerId, isReconnect: boolean) => void} [onPlayerJoined]
 *   玩家加入/重連後的遊戲補發邏輯
 *
 * @property {(roomId, playerId, reason) => void} [onPlayerLeft]
 *   玩家離開/被踢後的遊戲跳過邏輯
 *
 * @property {(socket, roomId, playerId) => void} [onSpectatorUpgraded]
 *   觀戰者升為玩家後通知遊戲補發角色等
 */
```

### 7.11 房間封存

- **觸發**：所有連線（含觀戰者）都離線，且 **30 分鐘**內無人重連
- **行為**：清除 in-memory cache；`rooms.status` 更新為 `archived`
- **重連嘗試**：收到 `room:error { code: ROOM_ARCHIVED }`，前端引導回大廳

## 9. Testing Strategy

手動測試場景：

**基本流程**
- [ ] 建立房間 → 收到 `room:join-ack { type: 'player' }`
- [ ] 加入房間（大廳）→ 其他人收到 `room:player-joined`
- [ ] 加入不存在房間 → `room:error ROOM_NOT_FOUND`
- [ ] 加入已滿房間 → `room:error ROOM_FULL`
- [ ] 暱稱重複 → `room:error NICKNAME_TAKEN`
- [ ] 加入已封存房間 → `room:error ROOM_ARCHIVED`

**斷線重連**
- [ ] 玩家斷線 → 其他人收到 `room:player-offline`
- [ ] 玩家重連 → 收到 `room:join-ack { type: 'reconnect' }` 含完整狀態
- [ ] Server 重啟後重連 → 從 DB 重建 cache
- [ ] 默契傳聲筒重連 → 從 `character_storm_sessions` 還原角色/quota

**觀戰模式**
- [ ] 默契傳聲筒單輪中加入 → `room:join-ack { type: 'spectator' }`；顯示觀戰 UI
- [ ] revealing 空檔 → 觀戰者收到 `room:spectator-can-join`；升級成功
- [ ] 懂我再說 guessing 中加入 → type: 'player'；本輪猜測 disabled

**踢人**
- [ ] 房主踢人（人數充足）→ 被踢者收到 `room:kicked`；遊戲繼續
- [ ] 踢後人數不足 → `room:error KICK_BELOW_MIN` 含原因
- [ ] 非房主踢人 → `room:error FORBIDDEN`

**房主轉移**
- [ ] 主動轉移 → 全房間收到 `room:host-changed`
- [ ] 房主離開 → 自動轉移
- [ ] 房主斷線 30 秒 → 自動轉移

**封存**
- [ ] 空房間 30 分鐘後嘗試重連 → `room:error ROOM_ARCHIVED`

## 10. Boundaries

### Always
- 所有房間操作先驗證 `socket.data.playerId`
- 踢人前驗證房主身份 + 最低人數保護
- 事件名稱使用 `events.js` 常數
- 觀戰者不收私密遊戲資訊
- DB 操作全在後端（service_role）

### Ask First
- 修改 `rooms`、`players`、`character_storm_sessions` schema
- 修改 DB RLS 設定

### Never
- 前端持有 service_role key
- 觀戰者收到被猜者的私密答案
- 空房間超過 30 分鐘不清理

## 11. Success Criteria

### Must-Have

- [ ] `server/room/` 模組被兩遊戲共用，無需各自重寫房間邏輯
- [ ] 玩家以相同 `playerId` 重連，取回完整狀態（含角色/進度）

- [ ] 房主可踢人（含最低人數保護），遊戲繼續
- [ ] 房主離線 30 秒後自動轉移

- [ ] 懂我再說遊戲中加入 → 本輪跳過猜測，下輪正常參與
- [ ] 空房間 30 分鐘後封存

- [ ] 觀戰者看到即時遊戲畫面（倒數、提示進度等公開資訊）
- [ ] 前端 `useRoom.js` 讓各遊戲 composable 不重複處理 `room:join-ack`
- [ ] 封存後自動清理 `character_storm_sessions` 歷史

## 12. 已確認決策

| 問題 | 決定 |
|------|------|
| 語言/框架 | 維持 Node.js + Socket.io + Vue 3 |
