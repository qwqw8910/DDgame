## Task 列表

### Task 1 ｜新增 `previewing_question` 中間狀態
**難度：⭐⭐⭐⭐（高）— 影響後端狀態機 + 前端所有玩家畫面**

**需求描述：**
目前流程：選主題 → `selecting_answer` → `guessing` → `revealing`

新流程：
```
選主題
  ↓
[新狀態: previewing_question]
  → 題目只私下傳送給被猜者（其他玩家看到「等待 XXX 確認題目…」）
  → 被猜者可以「換一題」（每輪限 2 次，消耗後無法再換）
  → 被猜者點「確認題目」後，廣播題目給所有人
  ↓
selecting_answer → guessing → revealing
```

**後端改動：**
- `rounds.status` 新增 `previewing_question` 狀態
- 新增 Socket 事件：`preview_question`（只傳給被猜者）
- 新增 Socket 事件：`swap_question`（被猜者發送，後端抽新題目，再傳給被猜者）
- 新增 Socket 事件：`confirm_question`（被猜者發送，後端廣播題目，進入 `selecting_answer`）
- 後端記錄每輪換題次數（`swap_count`，上限 2 次）

**前端改動：**
- 新增 `previewing_question` 畫面區塊：
  - 被猜者：顯示題目 + 換一題按鈕（顯示剩餘次數）+ 確認按鈕
  - 其他玩家：顯示「等待 XXX 確認題目中…」

### Task 2 ｜每輪隨機輪序（不重複直到所有人輪完）
**難度：⭐⭐（中低）— 後端邏輯調整**

**需求描述：**
- 每輪隨機選出一位「尚未當過被猜者」的玩家
- 所有人都輪過一次後，重新隨機排序開始下一輪循環

**後端改動：**

- 新增 `round_cycle` 計數，方便前端顯示「第幾輪循環」

### Task 3 ｜改進計分設計
**難度：⭐（低）— 純後端邏輯**

**需求描述：**
- 原本：猜對的玩家各 +1 分
- 新增規則：
  - 若本回合**所有人都猜對** → 被猜者（房主角色本輪）**+1 分**（獎勵「難以捉摸」）
  - 若本回合**所有人都猜錯** → 被猜者 **-1 分**（懲罰「太好猜」或特殊設計）

**後端改動：**
- 在 `revealing` 結算階段，統計猜對/猜錯人數
- 加入全對/全錯的條件分支，更新被猜者分數

### Task 7 ｜建立房間人數上限選擇（附差異說明）
**難度：⭐⭐（中低）— 前後端小改**

**需求描述：**
- 建立房間時可選擇人數上限（目前已有欄位 `max_players`，確認 UI 是否完整）
- 在 MD / UI 提示中說明不同人數上限的影響

**人數上限設定差異說明：**

| 上限人數 | 影響 |
|---------|------|
| 2～4 人 | 每輪猜的樣本數少，結果較隨機；遊戲節奏快 |
| 5～8 人 | 最佳體驗範圍，統計有意義，等待時間適中 |
| 9～12 人 | 等待所有人猜完的時間變長；揭曉時資訊量大，熱鬧但容易混亂 |
| 設定大但實際人少 | 遊戲照常進行，房間只是「保留位置」；不影響功能，但空位不會被填補 |

**後端改動：**
- 確認 `max_players` 欄位已正確寫入並驗證（建立時拒絕超過上限的玩家加入）

**前端改動：**
- 建立房間表單：人數上限選擇器 + 差異提示文字（tooltip 或灰字說明）

## 建議開發順序

| 優先 | Task | 理由 |
|------|------|------|
| 1️⃣ 最先 | **Task 9** — 顯示被猜者姓名 | 純前端、改動最小、效果立竿見影 |
| 2️⃣ | **Task 2** — 隨機輪序 | 純後端邏輯、不影響 UI，完成後其他 Task 都能在正確輪序下測試 |
| 3️⃣ | **Task 1** — previewing_question 狀態 | 核心流程改變，影響後端狀態機與前端多個畫面，需要穩定的輪序後再做 |
| 4️⃣ | **Task 7** — 人數上限選擇 | 大廳功能，早做可讓後續測試更靈活 |
| 5️⃣ | **Task 8** — 顯示誰已選/未選 | 與 Task 6 配合效果更好，但可先獨立完成 |
| 6️⃣ | **Task 6** — 可更改答案 | 需要後端配合，Task 8 完成後體驗更連貫 |
| 7️⃣ | **Task 3** — 計分設計 | 邏輯簡單，後端改動小，隨時可插入 |
| 8️⃣ | **Task 4** — 揭曉倒數動畫 | 純前端，不阻擋其他開發 |
| 9️⃣ 最後 | **Task 5** — Emoji 反應系統 | 純錦上添花，對核心邏輯無依賴 |
